
`call` 指令会将返回地址推送到栈上，这会改变栈顶的位置。这里是一个简化的视图，显示了 `call read_disk` 指令后栈的状态：

```
+-----------------+
| Return Address  | <- ESP在call指令后
+-----------------+
| core_start_addr |
+-----------------+
| 0x01            |
+-----------------+
```

当进入 `read_disk` 函数时，栈的状态将如下：

```
+-----------------+
| Return Address  | <- ESP在进入read_disk函数后
+-----------------+
| core_start_addr | <- 栈指针+4
+-----------------+
| 0x01            | <- 栈指针+8
+-----------------+
```

当在 `read_disk` 函数中执行 `push ebp`，它会将当前的EBP值（也就是旧的基指针）推送到栈上，这又会改变栈的状态：

```
+-----------------+
| Old EBP Value   | <- ESP在push ebp指令后
+-----------------+
| Return Address  |
+-----------------+
| core_start_addr | <- 新EBP值+8
+-----------------+
| 0x01            | <- 新EBP值+12
+-----------------+
```

然后，执行 `mov ebp, esp` 会设置新的基指针为当前的栈顶。

所以，在函数内部获取参数时，你应该使用如下偏移：

```assembly
mov eax, [ebp + 8]  ; 获取 core_start_addr
mov ecx, [ebp + 12] ; 获取 0x01
```

 `read_disk` 函数结束时，`pop ebp` 将恢复旧的基指针（EBP寄存器），而 `ret` 会从栈上弹出返回地址并跳转回调用者的下一条指令。调用者必须负责清理之前推送的参数：

```assembly
add esp, 8      ; 清理堆栈，因为我们推入了两个参数
```

这样，就保持了堆栈平衡。



在 x86 架构的保护模式下，门描述符（Gate Descriptors）用于控制任务和中断处理程序的访问权限，以及提供不同特权级（Ring 0 至 Ring 3）代码和数据之间的调用机制。门描述符在中断描述符表（IDT）和局部（LDT）或全局描述符表（GDT）中定义，并且有几种类型，包括调用门（Call Gates）、中断门（Interrupt Gates）、陷阱门（Trap Gates）和任务门（Task Gates）。以下详细描述了这些不同类型的门描述符及其用例。

### 调用门（Call Gates）

调用门用于在不同特权级的代码段之间进行控制转移。当一个程序需要执行更高特权级（更低的数字，如Ring 0是最高特权级）的代码时，它可以通过调用门来实现。

- **用途**：允许低特权级的代码（如用户模式代码）安全地调用高特权级的服务（如操作系统内核提供的服务）。
- **机制**：当通过调用门调用时，处理器会检查当前特权级（CPL）与调用门的特权级（DPL）以及目标代码段的特权级（RPL），并在适当的时候进行栈切换以保持特权级的一致性。

### 中断门（Interrupt Gates）和陷阱门（Trap Gates）

中断门和陷阱门都用于中断和异常的处理。它们之间的主要区别在于中断门在进入中断服务例程时会自动禁止进一步的中断，而陷阱门则不会。

- **中断门**：
  - **用途**：处理外部中断和内部异常，如硬件中断。
  - **机制**：在通过中断门调用中断服务例程（ISR）时，处理器会清除 `IF` 标志，禁止其他中断，避免中断嵌套。
  
- **陷阱门**：
  - **用途**：用于调试和一些内部处理，允许中断嵌套。
  - **机制**：当通过陷阱门进入ISR时，`IF` 标志不会被清除，允许中断重入。

### 任务门（Task Gates）

任务门提供了一个机制，通过它可以实现任务之间的切换。每个任务都有一个任务状态段（TSS），定义了任务的上下文信息。

- **用途**：允许处理器保存当前任务的状态，并加载新任务的状态，从而实现任务切换。
- **机制**：当任务切换时，处理器会自动保存当前任务的状态到其TSS中，并从目标任务的TSS中恢复其状态。

### 特权级（Privilege Levels）

在 x86 架构中，有四个特权级，从 Ring 0 到 Ring 3，其中 Ring 0 是最高特权级，通常用于操作系统内核，而 Ring 3 是最低的，用于用户模式应用程序。

- **Ring 0**：操作系统内核代码，有权访问所有硬件资源。
- **Ring 1 和 Ring 2**：通常不常用，有些操作系统会用于驱动程序或其他特殊的服务。
- **Ring 3**：用户模式代码，限制访问硬件资源。

特权级的设计确保了操作系统的核心部分不会被用户程序意外或恶意地干扰，同时也提供了一种机制，让用户程序通过控制的方式访问必要的系统资源。

综上所述，门描述符在操作系统中的作用是至关重要的，它们保证了系统的稳定性和安全性，同时也提供了一种灵活的方式，使得

不同特权级之间可以进行有控制的交互。



在 x86 体系结构中，当通过调用门发生特权级（privilege level）的转换时，处理器会执行一系列的检查和操作来确保转换的合法性和安全性。这些操作包括检查当前特权级（CPL）、调用门的特权级（DPL）和目标代码段的特权级（RPL），以及在必要的时候进行栈的切换。下面详细说明这个过程中所谓的“适当的时候”。

### 特权级检查

当通过调用门调用时，处理器首先检查：

1. **CPL（Current Privilege Level）**：当前正在执行代码的特权级。
2. **RPL（Requested Privilege Level）**：尝试访问段时的请求特权级，通常在选择子（segment selector）中给出。
3. **DPL（Descriptor Privilege Level）**：调用门描述符本身的特权级。

为了保证安全性，处理器要求：

- CPL 必须大于或等于调用门的 DPL。
- 调用门中指向代码段的选择子的 RPL 必须大于或等于调用门的 DPL。

如果这些条件不满足，处理器将触发一个一般保护异常（General Protection Fault，#GP）。

### 栈切换

如果检查通过，并且正在发生从低特权级（如Ring 3）到高特权级（如Ring 0）的转换，则处理器必须执行栈切换。这是因为每个特权级可能有自己的栈，这样做可以防止低特权级的代码影响高特权级的栈。适当的时候进行栈切换意味着以下几种情况：

- **从低特权级到高特权级的转换时**：处理器会从当前任务的任务状态段（TSS）中加载高特权级的栈指针（SS:ESP）。新的特权级对应的栈将用于代码执行期间的所有栈操作。
- **从高特权级返回到低特权级时**：通常是在中断或异常处理完成后，处理器将使用保存在栈中的先前特权级的栈指针来恢复到原来的特权级。

### 具体的栈切换过程

当栈切换发生时，以下步骤被执行：

1. 处理器从 TSS 中加载新的栈指针（SS 和 ESP）。
2. 旧的栈指针（SS 和 ESP）被保存到 TSS 中。
3. 处理器使用新的栈指针来继续执行代码。

这种机制确保了不同特权级的代码可以安全地交互，而不会相互干扰，从而保持系统的稳定性和安全性。



在操作系统中，TCB通常指的是Task Control Block，它是操作系统中用于管理进程或任务的数据结构。现在让我们比较一下TCB（Task Control Block）和TSS（Task State Segment）的区别。

### TSS（Task State Segment）

- **层级**：TSS是一个由x86体系结构定义的硬件层级的数据结构。
- **功能**：TSS主要用于任务切换，它保存了任务切换时需要保存和恢复的CPU状态信息，如寄存器状态、栈指针等。
- **存储**：它通常存储当前执行任务的状态，允许处理器在任务切换时能够保存当前任务的状态，并在之后恢复该状态。
- **使用**：在硬件任务切换中使用，虽然这在现代操作系统中不常用。

### TCB（Task Control Block）

- **层级**：TCB是一个由操作系统定义的软件层级的数据结构。
- **功能**：TCB用于保存和管理操作系统中任务的控制信息，如任务的状态、优先级、调度信息、指向任务的上下文（寄存器状态）、任务私有数据等。
- **存储**：它是进程或任务在操作系统中的表示，包含操作系统进行任务管理所需的所有信息。
- **使用**：在操作系统的任务调度和管理中使用，是现代操作系统中进行进程和线程管理的关键部分。

### 区别

尽管TSS和TCB都存储了有关任务状态的信息，但它们的用途和工作机制有显著不同：

- TSS直接与CPU硬件交互，它使得处理器能够支持多任务处理，并且在任务之间进行切换。
- TCB是操作系统级别的，用于在软件层面跟踪和管理任务，不直接由硬件使用，而是操作系统的调度器和其他管理组件使用。

在实践中，现代操作系统通常不使用硬件任务切换机制，而是采用自己的调度器和上下文切换机制，这些都是基于TCB实现的。因此，尽管TSS仍然存在于现代x86系统中（主要用于特权级别的变化，如系统调用），但是操作系统中的任务调度和状态管理则主要通过TCB来实现。
